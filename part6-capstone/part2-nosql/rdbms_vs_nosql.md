## Database Recommendation

For a healthcare startup building a patient management system, I would recommend MySQL as the primary database. Patient records, appointments, prescriptions, billing, and medical history are highly sensitive and require strong consistency, accuracy, and reliability. In healthcare, even a small inconsistency can affect treatment decisions, billing correctness, or regulatory compliance. MySQL follows ACID properties, meaning transactions are atomic, consistent, isolated, and durable. This makes it more suitable for systems where correctness and trust are more important than flexibility. A structured relational model is also useful because patient, doctor, appointment, and billing data are closely related and need clear relationships.

MongoDB is useful when the data structure changes frequently or when semi-structured data needs to be stored quickly. However, a patient management system is a high-trust system, so strong transactional control is more important than schema flexibility. From a CAP theorem perspective, healthcare systems should generally prioritize consistency over availability when there is a trade-off, because showing incorrect patient information is riskier than a temporary delay in access.

If the startup also needs a fraud detection module, then the recommendation changes slightly. I would still keep MySQL as the core system of record, but I would use MongoDB or another NoSQL database for fraud detection if it needs to process high-volume, fast-changing, or semi-structured data such as login activity, claim patterns, device metadata, or behavioral signals. Therefore, the best approach is hybrid: MySQL for the core patient management system and MongoDB as a supporting database for fraud analytics and detection.
